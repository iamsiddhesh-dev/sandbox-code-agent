# Sandbox Code Agent — Build Plan

**Orchestration model:** Fable 5
**What it is:** A natural-language → code → sandboxed execution → rendered result system. User asks for something ("plot the distribution of X", "clean this CSV", "write me a script that does Y"), the agent generates Python or JS, runs it inside a locked-down sandbox, recovers automatically when the code fails, and returns a chart, table, or working script.

**Why this project:** It demonstrates founding-engineer range end to end — reliable constrained code generation, real security boundaries around untrusted code execution, a self-healing generate/execute/retry loop with a hard failure ceiling, and clean typed output rendering. It's a mini version of what AI coding platforms do internally.

---

## Tech stack decisions (read before Phase 0)

- **Agent loop:** LangGraph. The generate→execute→retry cycle is a natural state graph with conditional edges (success → render, failure → retry, retries exhausted → give up gracefully). This is the right tool and a strong talking point on its own.
- **Sandbox:** **E2B** is the primary recommendation. Its free/hobby tier gives genuine microVM isolation (Firecracker), which is the honest answer to "real security boundaries," and the Python/JS SDK is purpose-built for exactly this (code interpreter sandboxes, file I/O, chart extraction). **Free-tier reality check:** E2B's free tier is credit/usage-metered and sandboxes have capped concurrent runtime and session length. That is fine for a portfolio project with bursty demo traffic, but you must treat it as metered, not unlimited.
  - **Free fallback if E2B doesn't fit:** run a **local Docker-based sandbox** (a hardened container: no network, read-only root FS, dropped capabilities, non-root user, CPU/memory/PID limits, `--tmpfs` scratch, execution timeout). This is $0 forever and arguably a *stronger* interview story because you built the isolation yourself instead of buying it. The plan is written so the sandbox is behind an interface — you can start on E2B and swap to Docker (or vice versa) without touching the agent loop. **Recommended path:** build the sandbox interface in Phase 2 with E2B as the default backend AND a Docker backend, so you can demo either and talk to both.
- **LLM for code generation inside the agent:** this is a runtime cost, separate from the orchestration model (Fable) used to *build* the project. Use a cheaper tier (Haiku 4.5 / Sonnet 5) as the code-gen model at runtime to keep the zero-cost goal realistic; see Phase 1.
- **Language scope — decide this consciously:** **Python is the primary target; JS is secondary.** Python has the richer analysis/charting story (pandas, matplotlib) and every example in this plan is Python-flavored. Build, eval, and demo Python end to end first; add the JS path in Phase 4 only once Python is green, and keep it to plain scripting + text/table output (skip JS charting). Do **not** try to hold both at parity from the start — that doubles the prompt-contract and renderer surface for very little portfolio gain. If you're short on time, shipping Python-only with an honest "JS backend is interface-ready, not built" note is a better outcome than two half-working languages.
- **Cost posture:** zero-cost tier everywhere feasible. E2B free tier, local Docker fallback, free LLM/dev credits, no paid hosting. **Two things are genuinely metered and must be watched, not assumed:** (1) runtime LLM tokens — a runaway repair loop is the realistic way to burn budget; (2) E2B sandbox credits/session-minutes. Budget guardrails for both are explicit tasks in Phase 3, not an afterthought.
- **Deployment posture:** the default deliverable is a **local demo + recorded GIF/screenshots**, which is a legitimate and common portfolio choice — a live public link is *not* required for this project to interview well. Hosting is scoped as **optional Phase 7**; skip it without guilt if the local demo is clean.

## The "model tier" column — what it means

Each phase names the tier to run **in Claude Code while you build that phase** (this is separate from the runtime code-gen model, which is decided in Phase 1). Reasoning: cheap tiers for scaffolding and glue, mid tiers for real implementation, top tiers only where subtle correctness or adversarial reasoning actually pays for itself.

⭐ = phase that doubles as a strong technical-interview talking point.

| 3 | **The Loop** ⭐ | Build the LangGraph generate→execute→retry cycle that recovers from failing code and cannot loop forever. | • Define LangGraph state: `request`, `code`, `exec_result`, `attempt`, `max_attempts`, `history`, `final_output`.<br>• Nodes: `generate` (LLM emits code), `execute` (sandbox runs it), `evaluate` (success vs failure classification), `repair` (feed stderr + failing code back to the LLM for a fix).<br>• Conditional edges: execute → success → render; execute → failure & attempts left → repair → execute; failure & no attempts → graceful give-up node.<br>• **Hard retry cap** (e.g. `max_attempts = 3`) enforced in state — the graph structurally cannot loop past it.<br>• Repair prompt: include the previous code, the exact stderr/traceback, and an instruction to fix only what broke.<br>• Distinguish failure classes: syntax/runtime error (retry-worthy) vs timeout/security-block (may warrant giving up faster).<br>• Log every attempt for observability/demo.<br>• Handle the "code succeeded but output doesn't match the envelope contract" case as a retryable failure.<br>• **Budget guardrails (the retry loop is how you burn money):** track cumulative input/output tokens and estimated cost in graph state; enforce a **per-request token/cost ceiling** that aborts the run even if `max_attempts` isn't exhausted (belt-and-braces with the attempt cap). ## Interview talking points (the ⭐ phases)

Rank order of what to lead with in a technical interview:

1. **Phase 3 — The Loop.** Termination guarantees, failure classification, self-repair from tracebacks. The clearest signal of engineering maturity: "my agent recovers from its own mistakes but provably cannot loop forever."
2. **Phase 2 — The Vault.** Real isolation with tests proving each boundary. "I don't trust the model's output, so here's the sandbox — and here are four tests proving it can't touch the network, escape the FS, exhaust memory, or run forever."
3. **Phase 5 — The Adversary.** Defense-in-depth: "the prompt is the first layer, the sandbox is the real guarantee — even a fully injected prompt can't exfiltrate because there's nothing to reach and nothing to steal."
4. **Phase 1 — The Prompt Contract.** Reliable structured output via a format envelope + few-shot. "Rendering is deterministic because the model writes to a contract, not free-form text."
5. **Phase 6 — The Gauntlet.** The data: measured reliability across request types. "Here are the numbers, not just a happy-path demo."

---

## Build order note

Phases are sequential and each unblocks the next: Groundwork → Prompt Contract → Vault → Loop → Render → Adversary → Gauntlet → *(optional)* Shipping Lane. Phases 2 (Vault) and 1 (Prompt Contract) are semi-independent and could be parallelized if you had two sessions, but the sequential order keeps the swappable-sandbox interface and the output envelope contract established before the Loop ties them together.

**Two-stage eval, by design.** Phase 1 evaluates the prompt **statically** (no sandbox exists yet — assert on output shape and parseability only). Phase 6 is the first **execution-based** eval. This is deliberate, not an oversight: don't try to run generated code in Phase 1, and do expect Phase 6 to surface contract violations that static checks couldn't catch.

**Project is "done" at Phase 6.** Phase 7 is opt-in. A clean local demo with recorded output is a complete, interview-ready deliverable — hosting adds cost and a public code-execution attack surface for marginal portfolio benefit.
